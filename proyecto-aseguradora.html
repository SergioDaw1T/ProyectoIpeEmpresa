Memoria Técnica del Proyecto

01. Visión General

La aplicación cubre toda la operativa de una aseguradora. No es solo un CRUD simple, son 20 entidades que se relacionan entre sí. Está dividida en varios módulos funcionales:

  • Administración: Gestión interna (roles, empleados, sucursales).
  • Comercial: Captación y asesoramiento (clientes, mediadores).
  • Pólizas: Núcleo del negocio (coberturas, bienes, documentos).
  • Facturación: Contabilidad de recibos y primas.
  • Siniestros: Flujo complejo de peritaciones e indemnizaciones.

Se diseñó un diagrama entidad-relación previo para asegurar la integridad de las claves foráneas en Oracle antes de proceder a la codificación.

02. Arquitectura Laravel

Uso del patrón Controller → Service → Repository → Model para garantizar un mantenimiento escalable.

app/Http/Controllers/RolController.php app/Services/Implementations/RolServiceImpl.php app/Repositories/Implementations/RolRepository.php app/Models/Rol.php

La lógica de negocio se extrajo a servicios para evitar controladores masivos, utilizando inyección de dependencias mediante interfaces registradas en el AppServiceProvider.

03. Frontend en Angular

Desarrollo modular con más de 80 componentes organizados por entidad (Alta, Búsqueda, Modificar, Detalles).

  • Formularios: Implementación mediante ngModel para agilidad en el desarrollo.
  • Routing: Estructura jerárquica con rutas hijas dentro de un dashboard centralizado.
  • Validación: Construcción manual de DTOs en el envío para asegurar el mapeo correcto con el backend.
const dto = new UsuarioDTO(); dto.username = form.value.username; dto.password_hash = form.value.password_hash;

04. Diseño y UX

Personalización visual profunda para alejarse de los estilos por defecto de Bootstrap.

  • Paleta: Degradados en azul marino (#0a1929) y acentos en Teal (#00bcd4).
  • Navegación: Header con menús desplegables categorizados por módulos de negocio.
  • Assets: Gestión de imágenes estáticas mediante configuración personalizada en angular.json.

05. Decisión: DTOs Planos

Optimización

Se eliminaron getters y setters innecesarios en TypeScript, optando por propiedades públicas inicializadas.

export class ClienteDTO { id_cliente: number = 0; nombre: string = ''; activo: string = 'S'; }

Esto redujo el código un 60% y facilitó la serialización JSON nativa.

06. Decisión: Consistencia API

Estandarización de nombres de rutas en plural (ej. /getUsuarios) tanto en Laravel como en los servicios de Angular para evitar errores 404.

Se realizó una refactorización masiva para asegurar que la comunicación entre ambos frameworks fuera totalmente simétrica.

07. Decisión: Selectores FK

Integridad

Implementación de carga dinámica de datos para claves foráneas. Antes de crear una póliza, el sistema carga automáticamente la lista de clientes, mediadores y productos disponibles.

Esto garantiza que el usuario nunca intente insertar un ID inexistente en la base de datos Oracle.

08. Decisión: Constructor Injection

Se optó por la inyección de dependencias clásica por constructor frente a la función inject().

Razones: Consistencia con otros lenguajes (Java/PHP) y mayor visibilidad de las dependencias requeridas por cada componente al inicio del fichero.

09. Decisiones Técnicas

  • snake_case: Sincronización total con los nombres de columna de Oracle.
  • Autenticación: Sistema de Login integrado con redirecciones automáticas.
  • Null Coalescing: Uso de ?? en PHP para manejar datos opcionales de la DB.
  • Limpieza: Eliminación de componentes redundantes para optimizar el bundle.

10. Retos y Soluciones

Oracle Case Sensitivity: Configuración de modelos con nombres en mayúsculas ($table = 'POLIZAS').

Tipos de Datos: Unificación de booleanos a strings ('S'/'N') para compatibilidad con VARCHAR2(1).

Angular Build: Ajuste de los límites de tamaño en angular.json debido al gran volumen de componentes.

Infraestructura: Recuperación de la base de datos tras un fallo crítico del servicio XE.
